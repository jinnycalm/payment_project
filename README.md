# PayPick (가제)
## 현 위치 기반 카테고리 탐색으로 주변 장소별 최적의 혜택을 실시간으로 매칭

## 📘 프로젝트 구조 (주요 파일)
```
pay_benefits/                 # 프로젝트 루트 (최상단)
├── client/                   # [프론트엔드 계층 - Vite + React]
│    ├── public/icons         # 아이콘
│    ├── src/                 # 실제 리액트 소스 코드
│    │   ├── main.tsx         # 리액트 시작점
│    │   ├── App.jsx          # 라우팅 관리 컴포넌트
│    │   ├── Home.jsx         # 랜딩 페이지
│    │   ├── MapPage.jsx      # 사용자 위치 기반 지도 표시 페이지
│    │   └── BenefitPage.jsx  # 선택한 업종에 대한 카드 혜택 추천 페이지

```
- 프론트엔드: React

### 🚀 프로젝트 개요

- **핵심 문제: '결제 시점의 정보 비대칭과 인지 부하'**
- **상세 내용:** 결제 수단(신용카드, 간편결제, 이벤트)은 폭증했으나, 각 수단의 **전월 실적 충족 여부, 잔여 할인 한도, 실시간 프로모션** 정보를 결제 직전 3초 내에 조합하기란 불가능에 가깝습니다. 사용자는 최적의 혜택을 알면서도 포기하거나, 잘못된 판단으로 혜택을 놓치는 '경제적 손실'을 매일 반복하고 있습니다.

---

### 💡 LLM / Agent 사용 이유

- **파편화된 정보:** 자산 관리 앱(뱅크샐러드 등)은 '과거 소비' 기록에 치중하며, 포털 검색은 '일반적인 혜택'만 보여줄 뿐 사용자의 **'현재 실재하는 한도'**를 반영하지 못합니다.
- **정적 데이터의 한계:** 카드사 앱은 이벤트(네이버페이 50% 적립 등)와 같은 타사 플랫폼의 실시간 변수를 고려하지 않습니다.
- **휴먼 에러:** 복잡한 예외 조항(백화점 입점 매장 제외, 상품권 제외 등)을 사용자가 일일이 기억하고 대조하기에는 한계가 명확합니다.

- **위의 한계점 때문에 LLM(실적, 혜택 관련 추론)과 Agent(작업 병렬 수행) 사용 필요**

### 📈 타겟 사용자

- **스마트 페이 마스터 (Smart Pay-Master):** 다중 카드/간편결제 최적화
- **합리적 자산 관리자:** 소액 절약의 시스템적 효율 추구
- **현장 결제 이벤트 사냥꾼:** 실시간 프로모션 활용

### ✨ 핵심 차별화

| 기존 앱 | PayPick |
|---------|---------|
| 과거 소비 분석 | **실시간 결제 최적화** |
| 정적 카드 혜택 | **RAG 기반 예외 조항 해석** |
| 단일 카드 추천 | **초 정밀 병렬 처리 엔진 (카드 + 간편결제)** |
| 혜택 정보 지연 | **실시간 이벤트 크롤링** |
| 일반적 조언 | **구체적 행동 지침 가이드** |

### 🧭 핵심 흐름도 (Pipeline)

1. **매장 조회:** 위치 기반 매장 식별.
2. **컨텍스트 수집:** 사용자의 보유 카드/실적/한도 데이터 로드.
3. **지능형 라우팅:** 각 카드 전담 RAG 일꾼과 실시간 이벤트 매칭 일꾼에게 작업 배분.
4. **2단계 병렬 분석:**

    **Level 1:** 카드사별 독립 분석.
    
    **Level 2:** 카드 내 [실적 판별 / 혜택 계산 / 예외 사항 검토] 동시 수행.
    
5. **데이터 합산 및 검증 (Map-Reduce):** 흩어진 분석 결과물을 JSON 형태로 통합하고 누락된 조항이 없는지 검수.
6. **최종 전략 수립 (Advice Node):** 사용자의 예상 결제 금액에 따른 '최종 승자(Winner)' 수단 결정.
7. **직관적 브리핑:** 근거(Evidence)와 함께 최적의 결제 수단을 가이드하고 결제 후 데이터를 즉시 최신화.
